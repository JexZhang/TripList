## 1. 项目背景

### 1.1. 现状

行迹是一个微信小程序，定位为"个人 / 家庭旅行攻略册"。当前实现已基于 Taro 4 + React 18 + TypeScript 跑通基础页面（单页内嵌 攻略 / 开销 / 清单 3 个 view，含 4 套主题、6 个清单模板），编译产物可在真机调试通过。

### 1.2. v1 重构动机

当前实现暴露了三类结构性问题：

1.2.1. **数据扩展性**：所有攻略横向排在首页一行，无法承载用户预期的多条攻略
1.2.2. **数据模型局限**：每天固定一个城市，无法表达"一日跨多城"的真实行程
1.2.3. **协作缺失**：纯本地 storage，无法分享或邀请家人协作

v1 的目标是：把单机原型升级为"可分享、可协作、可扩展"的攻略册，并在过程中重构数据模型、拆分页面结构、统一编辑交互。

### 1.3. 用户画像

- 主用户：制作攻略给家人看的"行程主理人"
- 协作者：被邀请参与编辑的家庭成员（人数通常 2-5）
- 接收者：只需查看攻略的家人朋友

## 4. 数据模型

云开发 NoSQL 文档数据库，3 个 collection。

### 4.1. `users` collection

```ts
{
  _id: 'openid',           // 主键 = 微信 openid
  _openid: 'openid',
  nickname: '张金池',
  avatarUrl: 'https://...',
  createdAt: timestamp,
  lastSeenAt: timestamp
}
```

### 4.2. `trips` collection（核心）

```ts
{
  _id: 'auto',
  _openid: 'openid',            // 创建者 / owner（自动注入）
  ownerOpenid: 'openid',        // 同 _openid，方便复合查询

  // 元数据
  name: '南京 · 金陵四日',
  pax: 4,
  startDate: '2026-05-29',
  endDate: '2026-06-01',
  destinations: [               // 多目的地
    { name: '南京', adcode: '320100', lat: 32.04, lng: 118.78 }
  ],

  // 协作
  collaborators: [
    {
      openid: 'xxx',
      nickname: '二姐',
      avatarUrl: 'https://...',
      role: 'editor',
      joinedAt: timestamp
    }
  ],

  // 内容 · 按天
  days: [
    {
      id: 'd1',                 // nanoid
      date: '2026-05-29',
      title: '三批抵宁 · 新街口集结',
      weather: {                // 可选，按 spots 第一个城市拉
        city: '南京',
        cityAdcode: '320100',
        temp: 28, low: 20, desc: '多云', icon: 'cloud',
        fetchedAt: timestamp
      },
      spots: [
        {
          id: 's1',              // nanoid
          type: 'spot',          // 'spot' | 'hotel' | 'meal' | 'transport' | 'arrive'
          time: '14:31',         // 可选
          name: '南京南站',
          city: '南京',
          adcode: '320100',
          lat: 31.97, lng: 118.80,
          price: 533,             // 可选
          note: '',
          // type='hotel' 额外字段
          nights: 1,
          // type='transport' 额外字段
          mode: '高铁',
          from: '北京南',
          to: '南京南'
        }
      ]
    }
  ],

  // 清单
  packing: [
    { id: 'p1', category: 'doc', label: '身份证', checked: false }
  ],

  // 时间戳
  createdAt: timestamp,
  updatedAt: timestamp,
  updatedBy: 'openid'           // 最后编辑者（协作显示用）
}
```

### 4.3. `share_tokens` collection

```ts
{
  _id: 'auto',
  token: 'xxx-32位随机',
  tripId: 'tripDocId',
  kind: 'readonly' | 'collab',
  createdBy: 'openid',
  createdAt: timestamp,
  expiresAt: timestamp,         // createdAt + 7 天
  usedBy: ['openid', ...]
}
```

### 4.4. 权限规则（云数据库 security rules）

- 4.4.1. **users**：仅本人读写自己那条
- 4.4.2. **trips**：
  - 读：`_openid == auth.openid || collaborators.openid contains auth.openid`
  - 写：`_openid == auth.openid || collaborators.openid contains auth.openid`
  - 删：仅 `_openid == auth.openid`（"删除"客户端会按 owner / collaborator 分支处理）
- 4.4.3. **share_tokens**：所有登录用户可读（按 token 查 trip 用）；仅 createdBy 可写

### 4.5. 关键决策

- 4.5.1. **trip 文档全量保存**：每次保存整个文档，不做 patch。简化逻辑，文档体积小（几 KB）不影响。
- 4.5.2. **`destinations` 是数组**：支持多目的地
- 4.5.3. **`spots[].city`**：每个 spot 自带城市；day 头部城市标签由 spots 聚合派生
- 4.5.4. **协作者 nickname / avatar 缓存在 trip 文档里**：避免每次显示都查 users 表

### 4.6. 删除语义

- 4.6.1. **只读分享接收方删除**：删除自己的副本文档（与原 trip 无关联）
- 4.6.2. **协作者删除**：客户端走 `db.collection('trips').doc(tripId).update({ collaborators: db.command.pull({ openid: self.openid }) })`，本人退出协作；owner 不感知
- 4.6.3. **owner 删除（含协作者的 trip）**：弹确认 `还有 N 位协作者将失去访问，确认删除？`，确认后真删
- 4.6.4. **owner 删除（无协作者）**：直接确认后删

## 5. 关键 UX 流程

### 5.1. 新建攻略

```
home → 点 [+ 新建攻略]
 │
 ▼
new-trip 表单（4 字段）
 │   ├─ 攻略名（文本）
 │   ├─ 日期（DatePicker 弹日历，多选起止）
 │   ├─ 目的地（DestinationPicker：高德城市搜索 + 多选 chips）
 │   └─ 人数（- 数字 +）
 │
 ▼ 点"创建"
 1. 客户端生成 days[] = 按日期范围每天 1 个空 day（含 date / 空 spots / 空 weather）
 2. 写云数据库 trips
 3. 异步：给每个 day 用 destinations[0] 的 adcode 拉一次高德天气，存进 day.weather
 4. 跳 trip?id=newId（默认进攻略 tab）
```

### 5.2. 添加 spot

```
trip · 攻略 view · 某天底部 [+ 添加地点]
 │
 ▼
SpotSearch 半屏
 ├─ 搜索框（高德 POI，debounce 300ms）
 ├─ 结果列表（按距离 / 相关度排）
 └─ 底部 [搜不到？手动添加 →]
 │
 ├─ 选了搜索结果
 │     → 立即 push 到 day.spots[]
 │       { id: uid(), type: 'spot', name, city, adcode, lat, lng,
 │         time: '', price: 0, note: '' }
 │     → 关闭 sheet，回攻略 view
 │
 └─ 点"手动添加"
       → 弹小输入框 [名字]
       → 立即 push（city/lat/lng 留空）
```

注意：**添加后所有字段都不必填**。用户后续点击该 spot 卡片进入 EditSpotSheet 补充。

### 5.3. 编辑 spot

```
点击 spot 卡片
 │
 ▼
EditSpotSheet 半屏
 ├─ 名称（点击 → 重开 SpotSearch 改地点）
 ├─ 时间（time picker）
 ├─ 类型（5 选 1）
 ├─ 价格（数字输入）
 ├─ 备注（多行文本）
 ├─ (type=hotel) 住几晚
 ├─ (type=transport) 方式 / 从 / 到
 └─ 底部 [删除地点]
 │
 ▼ 保存 / 删除 → 写 day.spots[] → 全量保存 trip
```

### 5.4. 加 / 删 day

- 5.4.1. **加 day**：day tab 末尾常驻 `[+]` tab，点击在 `days[]` 末尾追加 `{ id: uid(), date: lastDate + 1, ... }`
- 5.4.2. **删 day**：长按 day 标签 → 弹确认"删除 Day N？该日的 spots 一并删除"→ 确认后 `days.splice(idx, 1)`

### 5.5. 分享 flow

```
trip 页 → 头部 ⋯ → TripActionSheet
                       └─ 分享 → ShareTypeSheet
                                  ├─ 只读分享
                                  │   └→ 云函数 createShareToken('readonly')
                                  │      → 返回 token
                                  │      → wx.shareAppMessage({
                                  │          title: '行迹 · {tripName}',
                                  │          path: '/pages/share/index?token=xxx',
                                  │          imageUrl: 卡片图
                                  │        })
                                  │
                                  └─ 邀请协作
                                      └→ 云函数 createShareToken('collab')
                                         → 返回 token
                                         → 同样的 wx.shareAppMessage
```

### 5.6. 接收分享 flow

```
微信卡片 → 点击 → 跳 share?token=xxx
 │
 ▼ share 页加载
 1. 客户端拉 share_tokens 文档（按 token 查）
 2. 校验未过期、未被用过
 3. 拉 trips 文档（用 tokens.tripId）
 4. 渲染预览卡片 + 一个按钮：
    - token.kind='readonly' → [复制到我的攻略册]
    - token.kind='collab' → [加入协作]
 │
 ├─ 点"复制到我的攻略册"
 │   → 云函数 cloneTrip(sourceTripId, token)
 │     → 服务端 new trip 文档（_openid = recipient）
 │     → 把 source trip 的 days/packing/destinations 等内容深拷贝
 │     → collaborators 设为 []（独立副本）
 │     → 把 token 标 used
 │   → 客户端跳 home + toast"已复制"
 │
 └─ 点"加入协作"
     → 云函数 joinCollab(tripId, token)
       → 校验 token.kind === 'collab' 且未过期
       → 拉当前用户 nickname/avatar
       → 把当前用户 push 进原 trip 的 collaborators
       → 把 token 加入 usedBy（去重）
     → 客户端跳 trip?id=xxx + toast"已加入协作"
```

### 5.7. L1 协作同步

- 5.7.1. **进入 trip 页**：`db.collection('trips').doc(id).watch({ onChange })`
- 5.7.2. **本地编辑**：触发 reducer 更新本地 trip state → 500ms debounce → 写云数据库（全量替换）
- 5.7.3. **冲突策略**：写之前比对服务端 `updatedAt`；如果服务端 newer，先 fetch 最新版，做"以本地为基"的简单合并（last-write-wins on conflicting fields），toast 提示"X 也在编辑，已合并"
- 5.7.4. **远端推送**：watch onChange 回调时，如果本地有 pending 编辑则忽略（避免闪烁）；否则 setState 为远端版

| 云函数 | 输入 | 输出 |
|---|---|---|
| 6.2.1 `amap-poi-search` | `{ keyword, city? }` | `[{ name, address, city, adcode, lat, lng }]` |

### 6.3. 高德 API 接入

- 6.3.1. **key 仅在云函数环境变量**：`AMAP_KEY`，部署时设置，不进 git
- 6.3.2. **所有高德调用走云函数代理**：避免 key 泄漏、不受 referer 域名白名单限制
- 6.3.3. **天气缓存**：每 day 缓存 30 分钟，存在 `day.weather.fetchedAt`

## 9. 验收标准（v1 整体）

v1 上线时必须满足：

- 9.1. 用户能新建攻略并在自己手机上看到
- 9.2. 攻略可以加多个目的地，每天可以有跨城市的 spots
- 9.3. spot 增删改正常、字段都可选填、删除从详情页执行
- 9.4. 开销 view 自动从 spots 聚合，无需重复录入
- 9.5. 清单 view 可勾选、可从模板导入
- 9.6. 只读分享：A 发给 B，B 复制后独立编辑不影响 A
- 9.7. 协作邀请：A 发给 B，B 加入后能编辑；A 看到 B 的修改；B 看到 A 的修改；B 删除只移除自己
- 9.8. 微信开发者工具上传无阻断错误
