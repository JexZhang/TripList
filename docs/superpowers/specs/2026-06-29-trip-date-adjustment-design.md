# 1. 已有攻略日期调整设计

日期：2026-06-29
状态：用户已批准
范围：真实攻略页头部日期入口、ItineraryView 日期栏、TripStore 日期派生逻辑、行程整理面板

## 1.1. 背景

当前已有攻略的日期相关操作主要藏在 DayTabs 的长按菜单里。长按菜单同时承载前移、后移、移到最前、移到最后、删除等动作，发现性弱，且用户很难判断这些动作会改变日期、内容顺序，还是攻略天数。

本次设计目标是把“日期调整”和“行程安排整理”拆开：

1. 调整出发日期：只改变整段行程的起始日期，所有日期整体顺延，不增不删任何一天。
2. 整理行程：在整理模式中移动每天的行程安排，或删除某一天。
3. 添加天数：仍通过日期栏的 `+` 添加一天。

## 1.2. 设计原则

1. 不新增常驻“管理行程”按钮，避免攻略页控件继续变多。
2. 原长按日期菜单全部删除，不再用 action sheet 承载结构性操作。
3. 头部日期范围成为唯一常驻入口，用于打开“调整出发日期”面板。
4. 日期模型保持连续旅行日，不支持任意单日改成非连续日期。
5. 整理模式中日期槽位固定，用户拖动的是当天的行程安排内容。
6. 删除某天必须显式确认，并说明会删除多少个地点。

## 1.3. 用户交互

### 1.3.1. 攻略页常态

四个 TripHeader 主题都保留现有整体布局，只把现有日期范围改成可点击元素。

tegami、magazine、minimal 主题仍在 meta 行显示日期范围。日期范围视觉上使用各主题自己的可编辑样式，例如轻量边框、强调色或编辑提示，不新增额外按钮。

postcard 主题继续使用出发/抵达字段网格。出发字段作为可点击入口，抵达字段只展示自动派生的返回日期。

点击日期入口后打开底部面板。

### 1.3.2. 调整出发日期面板

面板标题为：

```text
调整出发日期
```

面板内容：

1. 出发日期 Picker。
2. 日期预览：

```text
行程日期将调整为 MM/DD → MM/DD
```

3. 行程安排分组：

```text
行程安排
调整顺序或删除某天
整理行程
```

4. 主按钮：

```text
确认调整
```

确认后只改变第一天日期，并按现有天数重新派生所有 `days[i].date`、`trip.startDate`、`trip.endDate`。不新增 Day，不删除 Day，不改变任何 Day 内的地点、预算、清单等内容。

### 1.3.3. 整理行程入口

进入整理行程有两个入口：

1. 在“调整出发日期”面板里点击“整理行程”。
2. 长按日期栏里的某个 Day 卡片。

长按不再弹出旧 action sheet。长按只作为快捷入口，直接进入整理行程，并高亮被长按的日期槽位。

### 1.3.4. 整理行程模式

整理页标题为：

```text
整理行程
```

页面顶部操作：

1. 左侧：取消
2. 中间：整理行程
3. 右侧：完成

整理列表采用“日期槽 + 行程安排卡”的结构：

1. 左侧固定日期槽：显示 `Day N` 和 `MM/DD`，不可拖动。
2. 右侧行程安排卡：显示该日期当前承载的行程安排摘要，可拖动。
3. 行程安排卡右侧提供删除入口。

示例：

```text
Day 1
06/30

伏见稻荷大社等 3 站
行程安排
删除
```

拖拽时只移动右侧行程安排卡到其他日期槽。日期槽位本身不移动。保存后 reducer 按当前槽位顺序写回 `days`，再保持连续日期。

### 1.3.5. 删除某天

点击整理模式中的删除入口时：

1. 如果只剩 1 天，不允许删除，提示至少保留 1 天。
2. 如果被删的 Day 有地点，弹确认。
3. 如果被删的 Day 没有地点，可以直接删除并轻提示。

有地点时确认文案：

```text
确认删除？
将会删除 X 个地点，删除后无法恢复。
```

按钮：

```text
取消
删除
```

删除后，剩余日期仍按连续日期重新派生。

## 1.4. 数据规则

### 1.4.1. 调整出发日期

新增 store action：

```ts
{ type: 'REBASE_DATES'; startDate: string }
```

行为：

1. 保留 `trip.days` 数量和顺序。
2. 将 `days[0].date` 设置为新的 `startDate`。
3. 调用现有连续日期派生逻辑，生成后续日期。
4. 同步更新 `trip.startDate` 和 `trip.endDate`。
5. 清空所有 Day 的 `weather` 字段，避免旧日期天气被误用。

### 1.4.2. 整理行程

整理模式提交的是新的 Day 内容顺序。实现上可以复用现有 `MOVE_DAY` 或新增批量排序 action。

重要约束：

1. 日期槽位固定，UI 中不可拖动日期。
2. 实际保存时，`days` 的内容顺序改变后，日期仍由当前首日连续派生。
3. 不支持任意给某一天指定非连续日期。

### 1.4.3. 删除某天

复用现有 `DELETE_DAY`，但入口改为整理模式里的显式删除。

约束：

1. `trip.days.length <= 1` 时禁止删除。
2. 有地点的 Day 删除前必须确认。
3. 删除后通过现有 `withSyncedDays` 保持 `startDate/endDate/days.date` 一致。

### 1.4.4. 添加一天

日期栏 `+` 行为保持不变，仍在末尾添加一个空白 Day。新增 Day 的日期由现有连续日期规则派生。

## 1.6. 错误处理与边界

### 1.6.1. 空行程

如果 `trip.days` 为空，调整出发日期面板禁用确认，并提示：

```text
请先添加一天行程
```

用户仍通过日期栏 `+` 创建第一天，避免日期调整面板隐式新增 Day。

### 1.6.2. 只剩一天

整理模式中只剩 1 天时，删除入口禁用或点击后提示：

```text
至少保留 1 天
```

### 1.6.3. 天气

调整出发日期、整理行程顺序、删除某天后，旧天气不再可信。实现时清空所有 Day 的 `weather` 字段，避免展示旧日期天气。

### 1.6.4. 只读模板

模板只读页仍不能展示编辑入口：

1. 不展示调整出发日期面板入口。
2. 不展示整理行程入口。
3. 不响应长按整理。

### 1.6.5. 协作同步

整理和日期调整都走现有 trip-store dispatch，保存仍由现有 500ms debounce 和 `update-trip` 云函数处理。远端协作者通过 watch 同步。

## 1.7. 验收标准

1. 点击真实攻略头部日期范围，打开“调整出发日期”面板。
2. 选择新出发日期后，预览展示新的起止日期。
3. 点击“确认调整”后，所有 Day 日期整体顺延，Day 数量和内容不变。
4. 日期栏长按不再出现旧 action sheet。
5. 长按某个 Day 后进入“整理行程”，并高亮该日期槽位。
6. 整理模式中日期槽位不可拖动，只有行程安排卡可拖动。
7. 整理后点击完成，行程内容顺序更新，日期仍保持连续。
8. 删除有地点的 Day 时，弹出“确认删除？”并显示将删除的地点数。
9. 删除空白 Day 不需要冗长确认。
10. 只剩 1 天时不能删除最后一天。
11. 只读模板页不出现任何日期调整或整理入口。
12. 四个主题的头部日期入口都保持各自设计风格。

## 1.8. 非目标

1. 不支持直接修改结束日期。
2. 不支持任意修改某一天为非连续日期。
3. 不在攻略页新增常驻“管理行程”按钮。
4. 不改变模板复制时选择出发日期的现有流程。
5. 不新增云函数。
