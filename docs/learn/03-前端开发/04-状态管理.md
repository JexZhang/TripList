# 第三章 · 状态管理

## 问题：状态越来越难管理

假设只用 `useState`，当多个组件需要同一份数据时就麻烦了：

```
TripPage
  ├── ItineraryView（需要 trip 数据）
  ├── BudgetView（需要 trip 数据）
  ├── PackingView（需要 trip 数据）
  └── MapView（需要 trip 数据）
```

如果在 TripPage 用 useState 管理 `trip` 数据，需要把它一层层传下去——这叫「Prop Drilling（属性钻探）」，很麻烦。

解决方案：**Context + Reducer**

---

## React Context：全局数据共享

Context 像一个「广播电台」，任何组件都可以「收听」：

```tsx
// 1. 创建 Context
const TripContext = createContext(null)

// 2. 用 Provider 包裹，提供数据
function TripProvider({ children }) {
  const [trip, setTrip] = useState(null)

  return (
    <TripContext.Provider value={{ trip, setTrip }}>
      {children}
    </TripContext.Provider>
  )
}

// 3. 任何子组件都可以取数据
function AnyDeepComponent() {
  const { trip } = useContext(TripContext)  // 直接取，不用传 props
  return <Text>{trip.name}</Text>
}
```

## Reducer：可预测的状态更新

`useState` 适合简单状态，但当状态很复杂（行册的 trip 有 days、spots、packing 等多层嵌套），更新逻辑容易出错。

`useReducer` 的思路：

```
当前状态 + 操作描述(Action) → Reducer 函数 → 新状态
```

类比：ATM 机
- 当前状态 = 余额 1000 元
- 操作 = `{ type: '取款', amount: 200 }`
- Reducer = 取款逻辑
- 新状态 = 800 元

```tsx
// 定义 Action 类型
type Action =
  | { type: 'ADD_SPOT'; dayId: string; spot: Spot }
  | { type: 'DELETE_SPOT'; dayId: string; spotId: string }
  | { type: 'UPDATE_TRIP'; patch: Partial<Trip> }

// Reducer：根据 Action 返回新状态
function tripReducer(state: State, action: Action): State {
  switch (action.type) {
    case 'ADD_SPOT':
      return {
        ...state,
        trip: {
          ...state.trip,
          days: state.trip.days.map(day =>
            day.id === action.dayId
              ? { ...day, spots: [...day.spots, action.spot] }
              : day
          )
        }
      }
    // 其他 case...
  }
}

// 使用
const [state, dispatch] = useReducer(tripReducer, initialState)

// 触发操作
dispatch({ type: 'ADD_SPOT', dayId: 'day-1', spot: newSpot })
```

## 行册的 trip-store 解析

`src/store/trip-store.tsx` 是整个应用最核心的文件，管理攻略的所有状态。

### 状态结构

```typescript
interface State {
  trip: Trip | null   // 当前攻略数据（主要）
  loading: boolean    // 是否加载中
  error: string | null // 错误信息
}
```

### 支持的 Action（操作类型）

```typescript
type Action =
  | { type: 'SET_TRIP'; trip: Trip }              // 设置整个攻略（初次加载）
  | { type: 'UPDATE_TRIP'; patch: Partial<Trip> } // 更新攻略基本信息
  | { type: 'ADD_DAY'; day: Day }                 // 添加一天
  | { type: 'DELETE_DAY'; dayId: string }         // 删除一天
  | { type: 'UPDATE_DAY'; dayId: string; patch: Partial<Day> } // 更新某天
  | { type: 'ADD_SPOT'; dayId: string; spot: Spot }            // 添加地点
  | { type: 'UPDATE_SPOT'; dayId: string; spotId: string; patch: Partial<Spot> } // 更新地点
  | { type: 'DELETE_SPOT'; dayId: string; spotId: string }     // 删除地点
  | { type: 'ERROR'; message: string }            // 错误处理
```

### Reducer 的核心逻辑（简化版）

```tsx
function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'SET_TRIP':
      return { ...state, trip: action.trip, loading: false }

    case 'ADD_SPOT': {
      const { dayId, spot } = action
      return {
        ...state,
        trip: {
          ...state.trip!,
          days: state.trip!.days.map(day =>
            day.id === dayId
              ? { ...day, spots: [...day.spots, spot] }  // 在指定天加入地点
              : day
          )
        }
      }
    }

    case 'DELETE_SPOT': {
      return {
        ...state,
        trip: {
          ...state.trip!,
          days: state.trip!.days.map(day =>
            day.id === action.dayId
              ? { ...day, spots: day.spots.filter(s => s.id !== action.spotId) }
              : day
          )
        }
      }
    }

    // ...其他 case
  }
}
```

## 「不可变更新」：为什么要 `...spread`

这是 React 的重要原则：**不直接修改原状态，而是创建新的状态对象**。

```tsx
// ❌ 错误：直接修改
state.trip.name = '新名字'
// React 检测不到变化，界面不更新

// ✅ 正确：创建新对象
return {
  ...state,           // 展开旧 state 的所有属性
  trip: {
    ...state.trip,    // 展开旧 trip 的所有属性
    name: '新名字'    // 覆盖 name 这一个属性
  }
}
```

`...` 是「展开运算符」，`{ ...obj, key: newValue }` = 复制 obj 的所有属性，然后用 `newValue` 覆盖 `key`。

## 自动保存：防抖 + useEffect

trip-store 有一个很重要的副作用：监听 trip 数据变化，自动保存到云端。

```tsx
// 上一次保存的「指纹」，用来检测是否真的有变化
const lastSavedRef = useRef<string>('')
const pendingRef = useRef(false)  // 是否是自己刚做的改动（实时同步用）

useEffect(() => {
  if (!trip || isSeedTrip) return  // 示例攻略不保存

  const fingerprint = JSON.stringify({
    days: trip.days,
    packing: trip.packing,
    name: trip.name,
    pax: trip.pax,
  })

  if (fingerprint === lastSavedRef.current) return  // 没有实质变化，跳过

  // 500ms 防抖：等用户停止操作再保存
  const timer = setTimeout(() => {
    lastSavedRef.current = fingerprint
    pendingRef.current = true  // 标记：这是自己的改动

    updateTrip(trip._id, { days: trip.days, packing: trip.packing, ... }, openid)
      .finally(() => {
        pendingRef.current = false
      })
  }, 500)

  return () => clearTimeout(timer)  // 清理：每次 trip 变化都重置定时器
}, [trip])
```

## 实时同步：数据库监听

```tsx
useEffect(() => {
  if (!tripId || isSeedTrip) return

  const db = Taro.cloud.database()
  const watcher = db.collection('trips').doc(tripId).watch({
    onChange(snapshot) {
      const updated = snapshot.docs[0]
      if (!updated) return

      // 如果是自己刚做的改动，忽略（避免重复更新）
      if (pendingRef.current) return

      dispatch({ type: 'SET_TRIP', trip: updated })
    },
    onError(err) {
      console.error('watch error', err)
    }
  })

  // 组件卸载时关闭监听
  return () => watcher.close()
}, [tripId])
```

`pendingRef.current` 是一个巧妙的设计：自己保存时设为 `true`，这样收到数据库的「回声」（自己改动触发的通知）时不会重复更新。

## Provider 的使用方式

```tsx
// src/store/trip-store.tsx 提供了两个东西
export const TripProvider  // 包裹组件，提供状态
export const useTripStore  // Hook，取状态

// 使用（src/pages/trip/index.tsx）
export default function TripPage() {
  const router = useRouter()
  const tripId = router.params.id

  return (
    // 用 TripProvider 包裹，所有子组件都能访问攻略状态
    <TripProvider tripId={tripId} openid={openid}>
      <TripBody />
    </TripProvider>
  )
}

// 在任意子组件中取数据
function ItineraryView() {
  const { state, dispatch } = useTripStore()
  const trip = state.trip!  // 攻略数据

  return <Text>{trip.name}</Text>
}
```

## me-store：用户状态

还有一个更简单的 store，管理当前用户信息：

```tsx
// src/store/me-store.tsx
const MeContext = createContext(null)

export function MeProvider({ children }) {
  const [openid, setOpenid] = useState('')
  const [nickname, setNickname] = useState('')

  useEffect(() => {
    // 登录时获取用户信息
    cloud.ensureUser().then(({ openid, nickname }) => {
      setOpenid(openid)
      setNickname(nickname)
    })
  }, [])

  return (
    <MeContext.Provider value={{ openid, nickname }}>
      {children}
    </MeContext.Provider>
  )
}

export function useMe() {
  return useContext(MeContext)
}
```

## 总结对比

| 方式 | 适用场景 | 行册里的例子 |
|------|---------|------------|
| `useState` | 组件内部状态 | `searchOpen`, `keyword` |
| `useReducer` | 复杂状态，多种更新操作 | `trip-store` 的攻略数据 |
| `Context` | 多组件共享数据 | `TripProvider`, `MeProvider` |

**规律**：
- 简单 → `useState`
- 复杂多操作 → `useReducer`
- 需要跨组件共享 → `Context`
