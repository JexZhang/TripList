# 第五章 · 行程编辑

行程编辑是行迹最核心的功能，本章深入分析它的实现细节。

---

## 功能概览

用户在攻略的「攻略」Tab 里可以：
- 切换日期（横向滚动的 Tab）
- 查看当天地点列表
- 添加地点（搜索 POI）
- 编辑地点信息（名称、类型、时间、价格、备注）
- 删除地点
- 添加/删除天数

---

## 状态设计

ItineraryView 有自己的局部状态：

```tsx
// src/views/ItineraryView/index.tsx
const [activeDayId, setActiveDayId] = useState<string>(trip.days[0]?.id || '')
const [searchOpen, setSearchOpen] = useState(false)
const [editSpot, setEditSpot] = useState<{ dayId: string; spot: Spot } | null>(null)
```

全局状态来自 `useTripStore()`：

```tsx
const { state, dispatch } = useTripStore()
const trip = state.trip!
```

## 添加地点：完整流程

### 1. 触发搜索

```tsx
<View className='itin-add-spot' onClick={() => setSearchOpen(true)}>
  + 添加地点
</View>

<SpotSearch
  open={searchOpen}
  defaultCity={activeDay.spots[0]?.city || trip.destinations?.[0]?.name}
  onClose={() => setSearchOpen(false)}
  onSelect={handleAddSpot}
/>
```

`defaultCity` 的逻辑：优先用当天已有地点的城市，没有就用攻略的第一个目的地。这样搜索结果默认在用户计划的城市里。

### 2. 搜索结果

```tsx
// SpotSearch 内部
useEffect(() => {
  if (!debouncedKw.trim()) { setResults([]); return }

  setLoading(true)
  cloud.searchPoi({ keyword: debouncedKw, city: defaultCity })
    .then(r => setResults(r.results))
    .catch(e => { console.error('[poi search]', e); setResults([]) })
    .finally(() => setLoading(false))
}, [debouncedKw, defaultCity])
```

### 3. 选择地点

```tsx
const useResult = (r: PoiResult) => {
  onSelect({
    name: r.name,
    city: r.city || undefined,
    adcode: r.adcode || undefined,
    lat: r.lat || undefined,
    lng: r.lng || undefined,
  })
  onClose()
}
```

### 4. 创建 Spot 对象

```tsx
// ItineraryView/index.tsx
const handleAddSpot = (info: SelectedSpotInfo) => {
  if (!activeDay) return
  const spot: Spot = {
    id: uid(),          // nanoid(12) 生成唯一 ID
    type: 'spot',
    name: info.name,
    city: info.city,
    adcode: info.adcode,
    lat: info.lat,
    lng: info.lng,
  }
  dispatch({ type: 'ADD_SPOT', dayId: activeDay.id, spot })
}
```

### 5. Reducer 更新状态

```tsx
// trip-store.tsx
case 'ADD_SPOT': {
  return {
    ...state,
    trip: {
      ...state.trip!,
      days: state.trip!.days.map(day =>
        day.id === action.dayId
          ? { ...day, spots: [...day.spots, action.spot] }
          : day
      )
    }
  }
}
```

### 6. 自动保存（500ms 防抖）

状态变化后，trip-store 里的 useEffect 检测到变化，500ms 后自动保存到云端。

## 编辑地点：EditSpotSheet

用户点击地点卡片打开 EditSpotSheet：

```tsx
<SpotCard key={s.id} spot={s} onClick={() => setEditSpot({ dayId: activeDay.id, spot: s })} />

<EditSpotSheet
  open={!!editSpot}
  spot={editSpot?.spot || null}
  defaultCity={...}
  onClose={() => setEditSpot(null)}
  onSave={patch => {
    dispatch({ type: 'UPDATE_SPOT', dayId: editSpot.dayId, spotId: editSpot.spot.id, patch })
  }}
  onDelete={() => {
    dispatch({ type: 'DELETE_SPOT', dayId: editSpot.dayId, spotId: editSpot.spot.id })
  }}
/>
```

EditSpotSheet 内部用 `draft` 状态暂存编辑中的数据，只有点「保存」才真正更新：

```tsx
// 打开时复制当前 spot 数据到 draft
useEffect(() => {
  if (open && spot) setDraft({ ...spot })
}, [open, spot])

// 保存：把 draft 的内容 patch 到 store
const save = () => {
  onSave(draft)      // 通知父组件
  onClose()
}

// 在 draft 里修改，不影响 store 里的原始数据
const set = <K extends keyof Spot>(k: K, v: Spot[K]) =>
  setDraft(d => ({ ...d, [k]: v }))
```

这种「草稿」模式确保用户取消时数据回滚，不留残影。

## 地点类型：不同类型显示不同字段

`type` 决定显示哪些字段：

```tsx
// EditSpotSheet 里
const type = draft.type || 'spot'

{/* 住宿：显示住几晚 */}
{type === 'hotel' && (
  <View className='es-field'>
    <Text className='es-label'>住几晚</Text>
    <Input
      value={draft.nights != null ? String(draft.nights) : ''}
      onInput={e => set('nights', parseInt(e.detail.value, 10) || undefined)}
    />
  </View>
)}

{/* 交通：显示方式、起点、终点 */}
{type === 'transport' && (
  <>
    <Input placeholder='高铁 / 飞机 / 自驾...' value={draft.mode || ''} ... />
    <Input placeholder='起点' value={draft.from || ''} ... />
    <Input placeholder='终点' value={draft.to || ''} ... />
  </>
)}
```

## 管理天数

### 添加天数

```tsx
const addDay = () => {
  const last = trip.days[trip.days.length - 1]
  // 新天 = 最后一天的下一天
  const newDate = last
    ? dayjs(last.date).add(1, 'day').format('YYYY-MM-DD')
    : dayjs(trip.startDate).format('YYYY-MM-DD')

  const day: Day = { id: uid(), date: newDate, spots: [], weather: null }
  dispatch({ type: 'ADD_DAY', day })
  setActiveDayId(day.id)  // 自动切换到新天
}
```

### 删除天数（长按触发）

```tsx
const longPressDay = async (dayId: string, dayNo: number) => {
  const res = await Taro.showModal({
    title: `删除 Day ${dayNo}?`,
    content: '该日的所有 spots 一并删除',
    confirmText: '删除',
    confirmColor: '#c43d3d',  // 红色确认按钮
  })
  if (res.confirm) {
    dispatch({ type: 'DELETE_DAY', dayId })
    // 如果删的是当前激活的天，切换到第一天
    if (activeDayId === dayId) {
      setActiveDayId(trip.days.find(d => d.id !== dayId)?.id || '')
    }
  }
}
```

## uid()：生成唯一 ID

行迹用 `nanoid` 库生成 12 个字符的随机 ID：

```typescript
// src/utils/id.ts
import { nanoid } from 'nanoid'

export function uid(): string {
  return nanoid(12)
}

uid()  // → "V1StGXR8_Z5j" 类似这样
```

为什么不用数字序号（1, 2, 3...）？
- 多人协作时，两个人同时添加地点，都生成 ID=4，就冲突了
- 随机 ID 几乎不可能冲突（12位字符，可能性有 62^12 ≈ 3万亿亿种）

## 小结

行程编辑的核心链路：

```
用户操作
    ↓
dispatch(action)
    ↓
Reducer 更新 trip 状态（不可变更新）
    ↓
React 重新渲染界面
    ↓
useEffect 检测到变化（防抖 500ms）
    ↓
cloud.updateTrip() 保存到数据库
```

关键设计：
1. **草稿模式**：EditSpotSheet 用 `draft` 暂存，保存时才提交
2. **防抖**：500ms 后批量保存，不是每次操作都请求
3. **nanoid**：生成唯一 ID，支持多人协作不冲突
4. **不可变更新**：所有状态修改都创建新对象
