# 第五章 · 用户登录

## 微信的「无感登录」

行册没有注册页面、没有填用户名密码，打开直接就能用。这是因为微信小程序有内置的身份体系。

### 传统 App 的登录

```
用户 → 注册（填邮箱/手机号） → 登录（输密码）→ 服务器验证 → 颁发 Token → 使用
```

### 微信小程序的登录

```
用户打开小程序 → 微信自动提供 openid → 直接使用
```

用户感知不到任何登录步骤。

---

## openid 是什么（再深一层）

```
微信服务器
    │
    ├── 用户张三 ──────── openid: oXXXX_aaaaa（在行册）
    │                                        oYYYY_aaaaa（在另一个小程序）
    │
    └── 用户李四 ──────── openid: oXXXX_bbbbb（在行册）
```

- 同一个用户在同一个小程序：openid **永远不变**
- 同一个用户在不同小程序：openid **不同**（保护隐私）
- openid 是随机字符串，不是用户真实信息

---

## 行册的登录流程

```
1. 小程序启动（app.tsx）
   ↓
2. cloud.init()：初始化云环境
   ↓
3. 调用 ensure-user 云函数
   ↓
4. 云函数通过 cloud.getWXContext() 获取调用者的 openid（由微信保证，无法伪造）
   ↓
5. 检查数据库：有没有这个 openid 的用户记录？
   - 有：更新 lastSeenAt，返回用户信息
   - 没有：创建新用户记录，返回用户信息
   ↓
6. 前端收到 { openid, nickname, avatarUrl }
   ↓
7. 把 openid 存到 me-store（全局状态）
   ↓
8. 后续所有操作都携带 openid
```

## 代码追踪

### Step 1：小程序启动

```tsx
// src/app.tsx
function App({ children }) {
  useEffect(() => {
    // 初始化云环境
    // @ts-ignore
    wx.cloud.init({ env: 'cloud1-d3gb6mt7red446466', traceUser: true })
  }, [])

  return (
    <MeProvider>  {/* 包裹整个应用，提供用户状态 */}
      {children}
    </MeProvider>
  )
}
```

### Step 2：MeProvider 初始化

```tsx
// src/store/me-store.tsx
export function MeProvider({ children }) {
  const [openid, setOpenid] = useState('')
  const [nickname, setNickname] = useState('')

  useEffect(() => {
    // 调用云函数确保用户存在
    cloud.ensureUser('行册旅人', '').then(({ openid, nickname }) => {
      setOpenid(openid)
      setNickname(nickname)
    })
  }, [])

  return (
    <MeContext.Provider value={{ openid, nickname }}>
      {children}
    </MeContext.Provider>
  )
}
```

### Step 3：攻略页面单独再调用一次

```tsx
// src/pages/trip/index.tsx
// 攻略页面需要 openid 来确认权限（是否是 owner）
useEffect(() => {
  Taro.cloud.callFunction({
    name: 'ensure-user',
    data: { nickname: '行册旅人', avatarUrl: '' }
  }).then((r: any) => {
    if (r.result && r.result.openid) {
      setOpenid(r.result.openid)
    }
  })
}, [])
```

## 为什么 Trip 页面单独再调一次 ensure-user？

因为 Trip 页面需要知道当前用户是不是这个攻略的 owner（不同操作权限不同），而 me-store 里的 openid 获取可能有竞态条件（先后顺序问题）。Trip 页面自己再确认一次，保证有正确的 openid 再渲染。

```tsx
// 没有 openid 时，显示「登录中...」
if (!openid) return <View className='trip-empty'>登录中...</View>

// 有了 openid 才渲染完整页面
return (
  <TripProvider tripId={tripId} openid={openid}>
    <TripBody />
  </TripProvider>
)
```

## 用户信息的权限边界

微信小程序的用户信息分两类：

### 自动获取（无需授权）
- openid（由微信后台提供，前端都不知道，只有云函数能获取）

### 需要用户主动授权
- 昵称
- 头像

行册目前没有要求用户授权昵称和头像，所以用的是默认值「行册旅人」。如果未来要显示真实昵称，需要添加授权流程。

---

## 协作权限模型

行册有三种角色：

| 角色 | 条件 | 能做什么 |
|------|------|---------|
| Owner（创建者）| `trip._openid === openid` | 全部操作：编辑、删除、分享、重命名 |
| Collaborator（协作者）| `collaboratorOpenids.includes(openid)` | 编辑内容（地点、清单等）|
| Viewer（只读访问）| 通过 readonly 分享链接 | 只能看，需要复制才能编辑 |

```tsx
// src/pages/trip/index.tsx
const isOwner = t._openid === openid

// 根据权限决定菜单项
<TripActionSheet
  actions={isSeedTripId(t._id) ? ['copy'] : undefined}
  // undefined = 默认显示所有操作
  // ['copy'] = 只显示复制（示例攻略）
/>
```

```tsx
// 分享功能只有 owner 才能用
const prepareShare = async (kind: ShareKind) => {
  if (!isOwner) {
    Taro.showToast({ title: '仅 owner 可分享', icon: 'none' })
    return
  }
  // ...
}
```

## 安全考量

1. **openid 不可伪造**：只有微信云函数能获取真实 openid，前端无法伪造
2. **权限在服务端校验**：`update-trip` 云函数每次都验证当前用户是否有权限
3. **Token 有过期时间**：分享 Token 7 天后失效，避免永久有效链接泄露

---

## 小结

- 微信小程序无需传统注册/登录流程
- openid 是用户唯一标识，由微信保证真实性
- `ensure-user` 云函数负责在数据库里建立用户记录
- 行册有三个权限级别：Owner、Collaborator、Viewer
