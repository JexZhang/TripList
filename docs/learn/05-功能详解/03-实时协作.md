# 第五章 · 实时协作

行册最复杂的功能之一：多个用户同时编辑同一份攻略，所有人的界面自动同步。

---

## 核心问题

当你和朋友同时打开同一份攻略：

```
你的手机                    朋友的手机
┌─────────────┐            ┌─────────────┐
│ Day1: 玄武湖  │            │ Day1: 玄武湖  │
│ Day2: 夫子庙  │            │ Day2: 夫子庙  │
└─────────────┘            └─────────────┘
```

朋友添加了「南京博物院」，你的手机怎么知道要更新？

**答案**：数据库「watch」监听机制。

---

## 微信云数据库的 watch

```javascript
// 建立实时监听
const watcher = db.collection('trips').doc(tripId).watch({
  onChange(snapshot) {
    // 每次数据库里的数据变化，这里就会被调用
    console.log('数据变了！', snapshot.docs[0])
  },
  onError(err) {
    console.error('监听出错', err)
  }
})

// 取消监听（组件卸载时调用）
watcher.close()
```

## 行册的实时同步实现

### 完整代码（简化版）

```tsx
// src/store/trip-store.tsx
function TripProvider({ tripId, openid, children }) {
  const [state, dispatch] = useReducer(reducer, initialState)
  const pendingRef = useRef(false)  // 标记：是否是自己刚做的改动

  // ① 加载初始数据
  useEffect(() => {
    if (isSeedTrip(tripId)) {
      // 示例攻略从本地 JSON 加载
      dispatch({ type: 'SET_TRIP', trip: getSeedTrip(tripId) })
      return
    }

    // 真实攻略从数据库加载
    getTrip(tripId).then(trip => {
      if (trip) dispatch({ type: 'SET_TRIP', trip })
      else dispatch({ type: 'ERROR', message: '未找到攻略' })
    })
  }, [tripId])

  // ② 建立实时监听
  useEffect(() => {
    if (isSeedTrip(tripId)) return  // 示例攻略不监听

    const db = Taro.cloud.database()
    const watcher = db.collection('trips').doc(tripId).watch({
      onChange(snapshot) {
        const updated = snapshot.docs[0]
        if (!updated) return

        // 关键：如果是自己刚做的改动，忽略这次回调
        // （数据库保存后会回传通知，但我们已经在本地更新了界面）
        if (pendingRef.current) return

        // 是别人的改动，更新本地状态
        dispatch({ type: 'SET_TRIP', trip: updated })
      }
    })

    // 组件卸载时关闭监听
    return () => watcher.close()
  }, [tripId])

  // ③ 防抖自动保存
  useEffect(() => {
    if (!state.trip || isSeedTrip(tripId)) return

    const timer = setTimeout(async () => {
      pendingRef.current = true  // 标记：我要改数据了
      try {
        await updateTrip(state.trip._id, {
          days: state.trip.days,
          packing: state.trip.packing,
          name: state.trip.name,
          pax: state.trip.pax,
        }, openid)
      } finally {
        pendingRef.current = false  // 标记：我的改动保存完了
      }
    }, 500)

    return () => clearTimeout(timer)
  }, [state.trip])
}
```

## pendingRef：避免「回声」问题

这是最巧妙的设计，用来解决「回声」问题：

```
你添加了地点
    ↓
dispatch → 本地状态更新 → 界面立即显示新地点（好的体验！）
    ↓
500ms 后保存到数据库
    ↓
数据库变化 → watch 触发 onChange
    ↓
如果不处理：dispatch SET_TRIP → 又更新了一次 → 界面闪烁！
```

`pendingRef` 的解决方案：

```
你添加了地点
    ↓
dispatch → 本地状态更新 → 界面立即显示（✅）
    ↓
500ms 后：pendingRef.current = true
    ↓
保存到数据库
    ↓
数据库变化 → watch 触发 onChange
    ↓
检查 pendingRef.current === true → 跳过！（✅ 不重复更新）
    ↓
保存完成：pendingRef.current = false
```

## 朋友修改时的流程

```
朋友的手机：
  用户操作 → dispatch → 状态更新 → 500ms后 → 保存到数据库

你的手机：
  watch 监听到数据库变化
     ↓
  pendingRef.current 是 false（不是自己的改动）
     ↓
  dispatch({ type: 'SET_TRIP', trip: 朋友更新后的数据 })
     ↓
  界面自动更新
```

## 协作者加入流程

1. Owner 点「分享」→ 选「邀请协作」→ 生成分享链接
2. 分享链接发给朋友（微信分享或复制链接）
3. 朋友点链接 → 进入 `pages/share/index.tsx`
4. 点「加入协作」→ 调用 `join-collab` 云函数
5. 云函数验证 Token，把朋友加入 `collaborators` 和 `collaboratorOpenids`
6. 跳转到攻略详情页，现在可以一起编辑了

```tsx
// src/pages/share/index.tsx
const handleJoin = async () => {
  setLoading(true)
  try {
    const res = await cloud.joinCollab({ tripId, token })
    if (res.alreadyOwner) {
      Taro.showToast({ title: '你是该攻略的创建者', icon: 'none' })
    } else if (res.alreadyJoined) {
      Taro.showToast({ title: '你已经是协作者了', icon: 'none' })
    } else {
      Taro.showToast({ title: '已加入协作！', icon: 'success' })
    }
    Taro.redirectTo({ url: `/pages/trip/index?id=${tripId}` })
  } finally {
    setLoading(false)
  }
}
```

## 退出协作 vs 删除攻略

`smartDeleteTrip` 根据用户角色做不同操作：

```typescript
// src/utils/db.ts
export async function smartDeleteTrip(trip: Trip, openid: string) {
  if (trip._openid === openid) {
    // 是 owner → 删除整个攻略（所有人都不能访问了）
    await deleteTrip(trip._id)
  } else {
    // 是 collaborator → 只退出，攻略还在，其他人还能访问
    // 调用 leave-collab 云函数（或直接用 update-trip 移除自己）
    await updateTrip(trip._id, {
      collaborators: trip.collaborators.filter(c => c.openid !== openid),
      collaboratorOpenids: trip.collaboratorOpenids.filter(id => id !== openid),
    }, openid)
  }
}
```

## 可能的竞态问题

多人同时编辑存在「最后写入者获胜」（Last Write Wins）的问题：

```
你和朋友都打开攻略，同时编辑 Day1：

你：添加「玄武湖」     朋友：同时添加「鸡鸣寺」
         ↓                       ↓
你的版本：[玄武湖]    朋友的版本：[鸡鸣寺]
         ↓（500ms后保存）         ↓（500ms后保存，稍晚一点）
      数据库：[玄武湖]         数据库：[鸡鸣寺]（朋友的覆盖了你的！）
```

这是行册当前的局限性——没有解决多人同时编辑同一字段的冲突。

真正的实时协作（如 Google Docs）用「操作变换（Operational Transformation）」或「CRDT」算法解决冲突，但这复杂得多。对于行程规划这种场景，「最后写入者获胜」通常够用了。

---

## 小结

实时协作的技术实现：
1. **数据库 watch**：监听数据变化，收到更新时刷新界面
2. **pendingRef**：避免自己的改动被「回声」重复处理
3. **防抖保存**：减少数据库写入频率
4. **权限分级**：Owner/Collaborator 权限不同
5. **Token 机制**：安全的分享链接
