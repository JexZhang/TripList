# 第四章 · 数据库设计

## NoSQL vs SQL

很多人学数据库先学 SQL（结构化数据库，如 MySQL）。微信云数据库用的是 NoSQL（非结构化），两者有重要区别：

| 对比 | SQL | NoSQL（微信云数据库）|
|------|-----|-------------------|
| 数据格式 | 行和列（表格） | 文档（像 JSON）|
| 结构 | 严格，提前定义 | 灵活，随时可加字段 |
| 查询 | SQL 语句 | 链式 API |
| 适合场景 | 关系复杂的数据 | 层级嵌套数据 |

行册的 Trip 数据天然是「嵌套」的（攻略包含多天，每天包含多个地点），所以 NoSQL 更合适。

## 数据库集合

行册使用两个集合（相当于 SQL 的「表」）：

```
微信云数据库
├── trips          ← 攻略数据（核心）
└── share_tokens   ← 分享 Token（临时数据）
```

## 设计决策解析

### 为什么把 days/spots 全部嵌套在 trips 里？

**方案 A（行册的做法）**：全部嵌套
```
trips 集合
└── 一条 trip 文档里包含所有 days 和 spots
```

**方案 B（分表）**：
```
trips 集合（基本信息）
days 集合（每天一条记录）
spots 集合（每个地点一条记录）
```

行册选方案 A，原因：
- 一次查询就能拿到所有数据（减少网络请求）
- 「攻略」的数据是整体保存/加载的，不需要单独查某一天
- 数据量不大（一个攻略通常不超过 50 个地点）

**方案 A 的缺点**：如果攻略数据很大（比如 500 个地点），每次保存都要传整个 days 数组，效率低。

---

### 为什么有 collaboratorOpenids 和 collaborators 两个字段？

`collaborators` 存完整信息（头像、昵称等），但微信云数据库的查询不支持对嵌套对象数组做高效 `in` 查询。

`collaboratorOpenids` 是一个简单字符串数组，专门用来做快速查询：

```javascript
// 查询"我参与的攻略"
db.collection('trips').where({
  collaboratorOpenids: db.command.elemMatch(db.command.eq(OPENID))
})
```

这是一种「冗余字段」设计——存两份同样的数据，为了性能。

---

### 时间戳为什么存字符串而不是 Date？

行册用 ISO 8601 字符串（如 `"2024-05-28T08:00:00Z"`），原因：
- 微信云数据库的 `db.serverDate()` 可能在不同端有差异
- 字符串排序和时间排序结果一致（ISO 格式天然可排序）
- 前端用 `dayjs` 处理，支持各种格式

## 数据库查询语法

```javascript
const db = cloud.database()
const _ = db.command  // 查询操作符

// 精确匹配
db.collection('trips').where({ name: '南京五日游' }).get()

// 比较
db.collection('trips').where({ pax: _.gt(2) }).get()  // pax > 2
db.collection('trips').where({ pax: _.lte(4) }).get() // pax <= 4

// 数组包含
db.collection('trips').where({
  collaboratorOpenids: _.elemMatch(_.eq('oXXXX_bbbbb'))
}).get()

// OR 查询
db.collection('trips').where(_.or([
  { _openid: 'oXXXX_aaaaa' },
  { collaboratorOpenids: _.elemMatch(_.eq('oXXXX_aaaaa')) }
])).get()

// 排序
db.collection('trips').orderBy('updatedAt', 'desc').get()

// 分页
db.collection('trips').skip(10).limit(10).get()

// 更新：追加元素到数组
db.collection('trips').doc(tripId).update({
  data: {
    collaboratorOpenids: _.push('newOpenid'),  // 追加
    pax: _.inc(1),                             // 自增
  }
})
```

## 数据校验

云数据库本身不做字段校验，行册在云函数里手动校验：

```javascript
// update-trip 云函数：白名单校验
const ALLOWED_FIELDS = ['name', 'pax', 'startDate', 'endDate', 'destinations', 'days', 'packing']
const safeData = {}
for (const key of ALLOWED_FIELDS) {
  if (key in patch) safeData[key] = patch[key]
}
// 这样即使前端传来 { _openid: 'hacker' }，也不会被保存
```

TypeScript 的类型定义（`src/types/trip.ts`）在前端做第一道防线，云函数做第二道。

## 数据大小限制

微信云数据库的限制：
- 单个文档最大 **512KB**
- 每次查询返回最多 **100** 条记录
- 批量操作每次最多 **100** 条

行册的应对：
- `list-my-trips` 限制返回 100 条（`.limit(100)`）
- trips 里的嵌套数据（days/spots）远小于 512KB 限制

---

## 小结

- 行册用两个集合：`trips`（核心数据）、`share_tokens`（临时 Token）
- 全部嵌套设计：一个攻略文档包含所有天数和地点
- 冗余字段（`collaboratorOpenids`）加速查询
- 所有写操作通过云函数，自带权限校验
