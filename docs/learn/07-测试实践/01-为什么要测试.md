# 第七章 · 为什么要测试

## 一个真实的故事

你花了一周写好了「开销统计」功能，上线了，大家用得很开心。

两周后，你修改了 `Day` 数据结构加了个新字段。

又过了一周，有用户反馈：「开销统计的金额算错了！」

你去查，发现修改数据结构的时候不小心破坏了开销计算逻辑。但当时根本没注意，因为修改的是完全不同的功能……

**如果有测试**，修改数据结构时测试会立刻失败，你马上就知道哪里出问题了。

---

## 测试是什么

测试是代码验证代码的代码。

```javascript
// 被测试的函数
function fmtCurrency(n) {
  return '¥' + n.toLocaleString()
}

// 测试代码
test('fmtCurrency 应该格式化金额', () => {
  expect(fmtCurrency(1234)).toBe('¥1,234')   // ← 这个断言必须成立
  expect(fmtCurrency(0)).toBe('¥0')
  expect(fmtCurrency(1000000)).toBe('¥1,000,000')
})
```

运行测试：
- 所有断言都通过 → ✅ 绿色
- 有断言失败 → ❌ 红色，告诉你哪行哪个期望值不对

---

## 测试的价值

### 1. 捕获回归 Bug

「回归」= 改了 A，不小心破坏了 B。

有测试的项目：改了代码，立刻运行测试，如果有破坏立刻发现。

```bash
$ npm test

FAIL src/utils/format.test.ts
  ✓ fmtCurrency 正常金额 (2ms)
  ✗ fmtCurrency 零元  # ← 这里失败了！
    Expected: '¥0'
    Received: '¥'      # ← 你改了什么导致零元显示错了？
```

### 2. 让你敢于重构

「重构」= 改善代码结构，不改功能。

没有测试时：你不敢改代码，因为不知道会不会破坏东西。  
有测试时：改完了跑测试，绿色就代表功能没变，放心提交。

### 3. 文档作用

测试代码本身就是对功能的描述：

```javascript
describe('buildNewTrip', () => {
  it('应该按日期范围生成正确数量的天数', () => { ... })
  it('出发日期等于结束日期时只有1天', () => { ... })
  it('应该将 pax 默认设为 1', () => { ... })
})
```

看测试就知道这个函数应该做什么、不应该做什么。

## 测试的种类

| 类型 | 测什么 | 速度 | 例子 |
|------|--------|------|------|
| **单元测试** | 单个函数/模块 | 极快（毫秒）| `fmtCurrency(1234) === '¥1,234'` |
| **组件测试** | React 组件的渲染和交互 | 快（秒）| 「点击按钮后弹窗是否出现」 |
| **集成测试** | 多个模块合作 | 中（秒）| 「添加地点后 store 是否更新」 |
| **E2E 测试** | 完整用户流程 | 慢（分钟）| 「从创建攻略到添加地点到保存」 |

行迹目前侧重单元测试和组件测试，因为：
- 速度快，可以频繁运行
- 可以不依赖云开发环境

---

## 哪些代码适合测试

✅ **非常适合**：
- 纯函数（给定输入，返回固定输出）
- 工具函数（`fmtCurrency`, `fmtDate`, `uid` 等）
- Reducer（`tripReducer`）
- 业务逻辑计算（开销统计、打包清单计数等）

⚠️ **需要 Mock（模拟）**：
- 网络请求（需要模拟 API 响应）
- 云函数调用
- 定时器

❌ **较难测试**（行迹里先跳过）：
- 微信小程序特有的 API（地图、分享等）
- 真实的数据库操作

---

## 行迹的测试工具

```
Jest      — JavaScript 测试框架，运行测试
@testing-library/react — 测试 React 组件
```

安装（如果还没安装）：
```bash
npm install --save-dev jest ts-jest @types/jest
npm install --save-dev @testing-library/react @testing-library/jest-dom
```

配置 `jest.config.js`：
```javascript
module.exports = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  testMatch: ['**/*.test.ts', '**/*.test.tsx'],
}
```

---

## 测试文件放哪里

行迹的惯例：测试文件放在被测文件旁边，后缀 `.test.ts`：

```
src/utils/format.ts
src/utils/format.test.ts    ← 测试文件

src/store/trip-store.tsx
src/store/trip-store.test.tsx
```

---

## 小结

- 测试 = 防止改了 A 不小心破坏 B
- 单元测试最容易写，从工具函数开始
- 测试代码本身也是文档
